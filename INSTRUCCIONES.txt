## SCHEMA DE BASE DE DATOS (Prisma)

Implementa EXACTAMENTE este schema en `prisma/schema.prisma`:

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")       // Supabase connection pooler (puerto 6543) — para queries en runtime
  directUrl = env("DIRECT_URL")         // Supabase direct connection (puerto 5432) — para migraciones Prisma
}

enum Role {
  SUPER_ADMIN
  ADMIN
  MANAGER
  EMPLOYEE
}

enum AttendanceStatus {
  PRESENT
  ABSENT
  LATE
  HALF_DAY
  HOLIDAY
  ON_LEAVE
}

enum RequestStatus {
  PENDING
  APPROVED
  REJECTED
}

enum PermissionType {
  MEDICAL
  PERSONAL
  FAMILY
  OTHER
}

enum DayOfWeek {
  MONDAY
  TUESDAY
  WEDNESDAY
  THURSDAY
  FRIDAY
  SATURDAY
  SUNDAY
}

model User {
  id            String    @id @default(cuid())
  employeeId    String    @unique  // ID visual tipo EMP-001 — generado por admin
  dni           String    @unique  // DNI/NIE del trabajador — obligatorio, validado formato español
  name          String
  email         String    @unique
  password      String    // bcrypt hash — establecida por el admin al crear el usuario
  role          Role      @default(EMPLOYEE)
  department    String?
  position      String?
  phone         String?
  startDate     DateTime  @default(now())
  isActive      Boolean   @default(true)
  avatarUrl     String?
  
  // Salario base para nómina
  baseSalary    Float     @default(0)
  
  // Días de vacaciones disponibles
  vacationDays  Int       @default(22)

  schedule      Schedule? @relation(fields: [scheduleId], references: [id])
  scheduleId    String?

  attendances   Attendance[]
  permissions   Permission[]
  vacations     Vacation[]
  payrolls      Payroll[]
  
  // Quién aprobó solicitudes (manager)
  approvedPermissions Permission[] @relation("ApprovedBy")
  approvedVacations   Vacation[]   @relation("ApprovedBy")

  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
}

model Schedule {
  id          String         @id @default(cuid())
  name        String         // Ej: "Turno General", "Turno Mañana"
  description String?
  workDays    WorkDay[]
  users       User[]
  createdAt   DateTime       @default(now())
  updatedAt   DateTime       @updatedAt
}

model WorkDay {
  id          String    @id @default(cuid())
  schedule    Schedule  @relation(fields: [scheduleId], references: [id])
  scheduleId  String
  dayOfWeek   DayOfWeek
  startTime   String    // "09:00" formato HH:mm
  endTime     String    // "17:00" formato HH:mm
  isWorkDay   Boolean   @default(true)
  
  @@unique([scheduleId, dayOfWeek])
}

model Attendance {
  id            String           @id @default(cuid())
  user          User             @relation(fields: [userId], references: [id])
  userId        String
  date          DateTime         // Solo la fecha (sin hora)
  checkIn       DateTime?        // Fecha+hora de entrada
  checkOut      DateTime?        // Fecha+hora de salida
  status        AttendanceStatus @default(PRESENT)
  totalHours    Float?           // Calculado al hacer checkout
  lateMinutes   Int              @default(0) // Minutos de tardanza
  notes         String?
  isManualEdit  Boolean          @default(false) // Si admin lo editó
  editedBy      String?          // userId del admin que editó

  createdAt     DateTime         @default(now())
  updatedAt     DateTime         @updatedAt

  @@unique([userId, date])
  @@index([date])
  @@index([userId])
}

model Holiday {
  id          String    @id @default(cuid())
  name        String
  date        DateTime
  isNational  Boolean   @default(true) // Nacional vs empresarial
  isRecurring Boolean   @default(false) // Si se repite cada año
  country     String    @default("ES")
  
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}

model Permission {
  id            String         @id @default(cuid())
  user          User           @relation(fields: [userId], references: [id])
  userId        String
  type          PermissionType
  startDate     DateTime
  endDate       DateTime
  reason        String
  status        RequestStatus  @default(PENDING)
  approvedBy    User?          @relation("ApprovedBy", fields: [approvedById], references: [id])
  approvedById  String?
  approvedAt    DateTime?
  rejectionNote String?
  documentUrl   String?        // Para adjuntar justificante

  createdAt     DateTime       @default(now())
  updatedAt     DateTime       @updatedAt
}

model Vacation {
  id            String        @id @default(cuid())
  user          User          @relation(fields: [userId], references: [id])
  userId        String
  startDate     DateTime
  endDate       DateTime
  totalDays     Int           // Días laborables calculados
  status        RequestStatus @default(PENDING)
  approvedBy    User?         @relation("ApprovedBy", fields: [approvedById], references: [id])
  approvedById  String?
  approvedAt    DateTime?
  rejectionNote String?
  notes         String?

  createdAt     DateTime      @default(now())
  updatedAt     DateTime      @updatedAt
}

model Payroll {
  id              String    @id @default(cuid())
  user            User      @relation(fields: [userId], references: [id])
  userId          String
  period          String    // "2024-01" formato YYYY-MM
  baseSalary      Float
  workedDays      Int
  totalDays       Int       // Días laborables del mes
  absentDays      Int       @default(0)
  latePenalty     Float     @default(0)
  bonus           Float     @default(0)
  deductions      Float     @default(0)
  grossSalary     Float
  netSalary       Float
  hoursWorked     Float
  status          String    @default("DRAFT") // DRAFT, GENERATED, PAID
  notes           String?

  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt

  @@unique([userId, period])
}
```

## MÓDULOS A IMPLEMENTAR — DETALLE COMPLETO

### 1. AUTENTICACIÓN (NextAuth.js v5)

**REGLA FUNDAMENTAL: No existe registro público en ningún punto de la aplicación. Solo los usuarios con rol ADMIN o SUPER_ADMIN pueden crear nuevas cuentas desde el panel de administración de empleados. Los trabajadores reciben sus credenciales (email + contraseña inicial) directamente del administrador.**

**Archivo:** `auth.ts` en la raíz

- Provider: Credentials (email + password)
- Password: bcrypt con 12 rounds
- Session: JWT strategy
- Incluir en el token JWT: `id`, `name`, `email`, `role`, `employeeId`, `dni`
- Callbacks session y jwt para enriquecer el token
- Páginas personalizadas: `/login`

**Login page `/login`:**
- Formulario centrado, minimalista
- Logo/nombre del sistema en la parte superior
- Campos: Email y Contraseña
- Botón "Iniciar Sesión"
- Mensaje de error visible si credenciales incorrectas
- **Sin enlace de registro, sin "¿Olvidaste tu contraseña?", sin opción de crear cuenta**

**Cambio de contraseña:**
- Un empleado puede cambiar su propia contraseña desde su perfil (sección "Seguridad")
- Un admin puede resetear la contraseña de cualquier empleado desde el panel de gestión
- El campo de nueva contraseña siempre requiere confirmación

**Middleware `middleware.ts`:**
- Proteger todas las rutas bajo `/(dashboard)/`
- Redirigir a `/login` si no hay sesión
- RBAC por ruta:
  - `/employees/*` → solo ADMIN y SUPER_ADMIN
  - `/schedules/*` → solo ADMIN y SUPER_ADMIN
  - `/holidays/*` → solo ADMIN y SUPER_ADMIN
  - `/reports/*` → ADMIN, SUPER_ADMIN, MANAGER
  - `/payroll/*` → ADMIN, SUPER_ADMIN
  - `/permissions/*` → todos (empleados ven solo las suyas)
  - `/vacations/*` → todos (empleados ven solo las suyas)
  - `/attendance/*` → todos (empleados ven solo las suyas)
  - `/attendance/mark` → todos los empleados

```typescript
// POST /api/attendance/mark
// - Obtener userId del JWT
// - Verificar que no haya ya una entrada+salida para hoy
// - Si no hay registro hoy → crear con checkIn = now()
// - Si hay checkIn pero no checkOut → actualizar checkOut = now()

// - Calcular lateMinutes comparando checkIn con horario asignado

// - Si el día no es laborable → error "No es día laborable"
```


## DESPLIEGUE — VERCEL + SUPABASE

### Supabase (base de datos)
No se necesita ningún archivo de configuración local. La base de datos vive en la nube desde el inicio.

### Vercel (aplicación)
Crear archivo `vercel.json` en la raíz:
```json
{
  "buildCommand": "prisma generate && next build",
  "installCommand": "npm install"
}
```

El comando `prisma generate` debe ejecutarse siempre antes del build en Vercel para que el cliente Prisma esté disponible.

### Variables de entorno en Vercel
En el dashboard de Vercel → Settings → Environment Variables, añadir:
```
DATABASE_URL        → URL del connection pooler de Supabase (puerto 6543)
DIRECT_URL          → URL de conexión directa de Supabase (puerto 5432)
NEXTAUTH_SECRET     → Clave secreta (openssl rand -base64 32)
NEXTAUTH_URL        → https://tu-dominio.vercel.app  (o dominio personalizado)
```

**Importante:** `NEXTAUTH_URL` en Vercel debe ser la URL de producción exacta, incluyendo `https://`.

### Prisma con Supabase — configuración crítica
```prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")    // pooler — usado en runtime y Vercel
  directUrl = env("DIRECT_URL")      // direct — usado solo por prisma migrate
}
```

El `directUrl` es imprescindible porque Supabase PgBouncer (pooler) no soporta las transacciones que usa `prisma migrate`. Las migraciones siempre usan la conexión directa.
